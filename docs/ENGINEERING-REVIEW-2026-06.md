# tdoc Engineering Review — Final Report

## 1. Verdict

tdoc is **fundamentally sound but not yet at "best practice, no more patches."** The core abstractions (content-hash artifact identity, append-only event-log comments, R2 write-verify) are well-designed and the worst-sounding findings were correctly refuted (no broad CSRF, no KV namespace escape, no cross-tier hash drift, no comment-shape split-brain at the wire layer). The gap is twofold: (a) **the most failure-prone correctness machinery is entirely untested** — `reconcileAnchors`, the auth/upload/R2-verify pipeline, `legacyToEvents` migration, and the worker comment API have zero automated coverage, and there is no test runner or CI to even run what exists; and (b) a cluster of **real-but-recoverable defects** (KV lost-update races, append-order event folds, the local server binding 0.0.0.0 with no auth and CSRF-able publish, unbounded curl/spawn hangs). Several confirmed items are explicit patch-smells: helpers copy-pasted with "must stay identical" contracts, abandoned-rewrite dead code, and escaping applied per-spot instead of as one layer. None of these are P0 — there is no live data-corruption or RCE on the default path — but the combination of untested critical paths plus no CI means regressions in exactly this bug family will ship silently. Close the testing/CI gap and the structural-duplication gaps and tdoc reaches the bar.

## 2. Confirmed Findings by Severity

### P1 — Untested critical paths + reachable security holes (6)

**`reconcileAnchors` has zero tests** — `worker/worker.js:251-317` (coverage)
The publish-time anchor-rebinding heuristic — the single most failure-prone correctness path — is unconditionally executed at the upload handler (`:1297`) and persists mutated anchors via `META.put(cKey, after)` (`:1299`), so a mis-bind silently rewrites user data. Every branch (single-candidate adoption `:278`, tag-only fallback `:280-282`, lost/ambiguous marking `:303-313`) is untested; `comment-history.test.js` slices out adjacent fold logic but never touches it. Shipped specifically to fix commit `8ab5b15`'s anchor-loss bug, yet has no regression guard. *Confirmed by: testing.*
Fix: VM-sandbox unit test driving synthetic comment lists + `aidsInVersion` arrays across all six branches; assert append-only events stamped at publish version.

**Worker auth + upload→R2/KV pipeline untested** — `worker/worker.js:724-965, 1257-1300` (coverage)
The highest-risk untested surface: bearer compare (`:727`), device-flow session minting + OAuth token persistence (`:925-970`), `isOwnerSession` (`:62`), `stampAids` on attacker-shaped HTML (`:1268`), and the `r2_write_lost` head-verify (`:1278-1285`). That verify is **itself a patch** for a prior silent-write-loss bug (per the in-code comment) and now has no regression test — a refactor that reorders the verify reintroduces data loss undetected, and `tdoc-publish:316-322` keys the whole publish contract off `.ok`. No Miniflare/workerd harness exists anywhere. *Confirmed by: testing.* **Patch-smell: the `r2_write_lost` guard is an unprotected prior-bug fix.**
Fix: workerd/Miniflare handler tests with Map-backed KV and a configurably-dropping R2 stub; pin the `r2_write_lost` path explicitly.

**Local server accepts unauthenticated cross-origin mutating requests** — `server/server.js:29-34, 293-320, 328-330` (security)
`readBody()` JSON-parses any body regardless of Content-Type, every mutating route (comments + `/api/publish`) has no Origin/CSRF/Content-Type check, and `server.listen(PORT)` with no host arg binds `::` (all interfaces, empirically verified). A drive-by page can fire a CORS-simple `text/plain` POST to `localhost:7878/api/publish` — which spawns `bin/tdoc-publish` — forcing a private draft public or tampering with comment threads. Not RCE (slug regex + argv spawn), hence P1 not P0. *Confirmed by: codex (and corroborated by the security-source finding below).*
Fix: bind `127.0.0.1`, reject non-JSON on mutating routes, enforce Origin/CSRF, remove publish from the anonymous API.

**Fork/export banner injects raw comment text into HTML comments** — `worker/worker.js:822-852` (security)
`/fork` and `/export` build an HTML-comment banner interpolating raw `c.text`/`r.text` (`:844`, `:848`); only newlines are stripped, not `-->`. A commenter (any GitHub session, not owner) posts `--><script>…` to break out of the comment and inject live markup at the top of the served document (`finalHtml = banner + jsonBlock + bodyHtml`, `:885`), executing on the worker origin that holds OAuth/session cookies. No CSP anywhere. `safeJsonForScript` protects the JSON block but is never applied to the banner. *Confirmed by: codex.*
Fix: never put untrusted text in HTML comments — keep only the `safeJsonForScript`-escaped JSON, or render escaped visible text nodes.

### P2 — Real, recoverable correctness/security/resilience defects + structural patch-smells (28)

**Concurrency / data model:**

- **KV read-modify-write has no CAS/lock → lost updates** — `worker/worker.js:1003-1029, 1043-1056, 1086-1117, 1135-1171, 1194-1246, 1291-1300` (concurrency). Every mutating comment endpoint does `get → JSON.parse → mutate → put` on the single `comments:<slug>` blob; two interleaved writers each append to their own copy and the second `put` clobbers the first, defeating the documented append-only invariant. No DO, no CAS anywhere. Two source agents independently flagged this (`kv-lost-update-race`, `worker-kv-rmw-lost-updates`). *Confirmed by: correctness, codex.* Fix: serialize per-slug through a Durable Object.
- **Event log folded in physical append order** — `worker/worker.js:529-531` + write sites + `tdoc-publish:329-334` (correctness). Folds replay events in array order, but paths append events stamped at *older* versions (anchor_changed `:1051`, reconcile `:287`) after newer ones; a backdated event then wins the latest snapshot — reachable via the version picker (`overlay.js:2409`) re-anchoring while viewing an old version, and via republish. Two source agents flagged (`worker-event-fold-append-order`, `snapshot-event-order-assumes-append-time-monotonic`). *Confirmed by: codex, correctness.* Fix: sort by `(at_version, original_index)` before folding; reject backdated writes.
- **`reconcileAnchors` lost-marker is sticky forever** — `worker/worker.js:258-262, 299-314` (correctness). A `lost` anchor becomes the folded `snap.anchor`; the `:262` guard then skips it on every future publish, so a re-added identical artifact (same content-hash aid) never re-binds. Orphaned permanently despite target existing. *Confirmed by: correctness.* Fix: treat `lost` element anchors as re-bind candidates, not skips. (Note: the lost record carries fingerprint/label/fallback but **not** `aid` — recovery must be fingerprint-based.)
- **KV single-value unbounded growth** — `worker/worker.js:982-987, 1015-1019, 1170-1171, 1205-1246` (scalability). All events for a slug in one append-only value; reaction toggles append both add and remove, reconcile appends per-version-per-publish, nothing compacts. O(total_events) re-serialize per click, toward the 25MB cap. *Confirmed by: data.* Fix: shard per-comment or move to DO/D1; fold reactions to an `(emoji,user)` set; the lowest-effort high-value sub-fix is reaction-set folding + publish-time compaction.

**Security / auth:**

- **Local server 0.0.0.0 + agent-reply spoof + publish env leak** — `server/server.js:184, 197, 309` (auth boundary). `/api/agent/reply` hardcodes `login:'tdoc-agent'` from an unauthenticated body; `/api/publish` spawns with `env: process.env` (inherits Cloudflare creds) and echoes child stderr/stdout verbatim (`:315`); `readBody` has no size cap. Same missing-loopback-bind root as the codex finding above. *Confirmed by: security.*
- **Author-only checks bypassed on null author** — `worker/worker.js:1048, 1095, 1109` (broken access control). `if (target.author && target.author.login !== s.login)` short-circuits to "allow" when author is null. `ensureEventLog` (`:496`) produces `author: null` for legacy records, so any signed-in GitHub user (open device-flow, no allow-list) can delete/re-anchor null-author legacy comments. Only legacy records affected (current code always stamps author), hence P2. **Patch-smell: the flawed pattern is duplicated 3×.** *Confirmed by: security.* Fix: invert to deny — `canMutate(record, session, env)` helper, 403 unless owner login matches or `isOwnerSession`.
- **Comment endpoints pass unvalidated slug into `path.join`** — `server/server.js:140, 147, 188, 223, 239, 263` (security). Only `/api/publish` validates slug; the comment routes don't, so `slug=../secret` reads/writes/deletes `comments.json` outside `TDOC_DIR` (live-verified: read and write of out-of-root files). Constrained (fixed filename, JSON-only, no mkdir), hence P2. *Confirmed by: codex.* Fix: reuse the strict slug regex, resolve and reject path escapes.

**CLI / resilience:**

- **`--force` deletes existing doc before validating replacement** — `bin/tdoc-new:105-131` (correctness). `rm -rf "$DOC_DIR"` (`:108`) runs before the emptiness/`<body>` checks (`:124-132`); empty stdin, failed cp, or markdown-by-mistake permanently destroys all versions + comments. The `<body>` check exists *specifically* to catch anticipated markdown input, which now nukes the prior doc. *Confirmed by: codex.* Fix: stage in temp, validate, atomic rename with rollback.
- **Every curl lacks `--max-time`/`--connect-timeout`** — `bin/tdoc-publish:122,140,160,165,311` (+`tdoc-pull:35`, `tdoc-doctor:73,82`) (resilience). A stalled endpoint hangs publish forever; the browser modal (`overlay.js:1736`, bare `fetch` no AbortController) spins permanently because `server.js:309-322` only responds on child `close`. *Confirmed by: errors.* Fix: add timeouts to every curl; impose an outer spawn deadline server-side.
- **`/api/publish` spawn: no timeout, no `on('error')`, unbounded buffering** — `server/server.js:309-322` (resilience). Pairs with the curl finding; additionally a spawn `error` (EACCES) with no listener crashes the whole local server. *Confirmed by: errors.* Fix: add `on('error')`→500, SIGTERM/SIGKILL deadline→504, cap buffered output.
- **`set -e` + `upload_version` aborts publish before latest version uploads** — `bin/tdoc-publish:331-336, 299-323` (resilience). One transient 500 on an older version (e.g. `r2_write_lost`) kills the script before the newest version's HTML lands; meta advertises a version whose R2 object 404s. Reachable on multi-version republish. *Confirmed by: errors.* Fix: collect per-version results, retry transients, never let an older-version failure block the latest.
- **CF-API responses jq-parsed with no HTTP-status check** — `bin/tdoc-publish:122-130,140-155,160-175` (+`tdoc-doctor`) (error-handling). A 401 (expired OAuth) JSON envelope makes the subdomain check misfire "no subdomain claimed" and doctor report a false "not ready"; a narrow path can create a duplicate META namespace. *Confirmed by: errors.* Fix: capture `%{http_code}`, treat non-2xx as a distinct retry state, share logic with doctor.
- **`tdoc-new` never checks the ping loop succeeded** — `bin/tdoc-new:157-166, 192-193` (error-handling). On server-start failure the loop exhausts without `break`, then prints `LOCAL_URL` as the success contract; callers capturing the last line get a dead URL with exit 0. *Confirmed by: errors.* Fix: capture loop outcome, die with `.server.log` tail on failure.
- **`published.json` fields read with no validation** — `bin/tdoc-publish:240-250` (error-handling). Missing field → `jq -r` prints literal `null` → `https://null.null.workers.dev` + `Bearer null`; the first-run write (`:216`)+chmod (`:217`) is non-atomic, briefly world-readable. *Confirmed by: errors.* Fix: validate each field (non-empty, not `null`); write atomically (tmp+chmod+rename).

## 3. The Single Highest-Value Thing to Fix Next

**Stand up a test runner + CI and immediately use it to lock down `reconcileAnchors` and the worker auth/upload/R2-verify pipeline.**

This is one move because the two halves are mutually reinforcing: the no-runner/no-CI gap (`no-test-runner-or-ci`) means even existing tests rarely run green end-to-end, and the two P1 coverage holes (`reconcile-anchors-untested`, `worker-auth-upload-untested`) sit on exactly the code that silently mutates and persists user data and whose `r2_write_lost` guard is an unprotected prior-bug fix. Concretely: add a root `package.json` + `test/run.js` aggregator + a CI workflow running the pure/local-server tests with no Cloudflare creds; add the `reconcileAnchors` VM-sandbox unit test (six branches) and a workerd/Miniflare handler test pinning the `r2_write_lost` path and the 401/403 auth gates. This converts the largest cluster of "ships silently" risk into "fails the build," and makes the MEMORY rule "E2E before every push" actually executable — the precondition for "no more patches."

## 4. What Was Refuted, and Why (25)

The verification pass killed every finding whose alarming framing didn't survive the code. Highlights, so you can trust the precision of what remains:

- **CSRF on cookie endpoints** — refuted: `SameSite=Lax` does *not* send the cookie on cross-site POST/fetch, there is no GET-based mutation, and CORS is `*` *without* `Allow-Credentials`, so scripted cross-site requests are strictly safe. (The genuine local-server CSRF is a *different*, confirmed P1 — that one has no cookie/SameSite protection at all.)
- **Slug unvalidated → KV namespace escape / cross-doc walk** (filed twice) — refuted: slug is always *appended* after a constant prefix (`comments:`/`meta:`/`session:`), so it can't escape backward; R2 keys are flat opaque strings; the worker is single-tenant so a token holder already owns the namespace. Residual is P3 input-hygiene, not injection.
- **Unauth GET exposes all comments + logins** — refuted: this is documented product behavior (public shareable links, GitHub auth gates *writes* only); the proposed fix contradicts the product model.
- **Worker/server comment-model fork breaks the overlay / corrupts pull** (filed twice) — refuted: the event log is a worker-internal KV detail; the API *always* folds to the flat shape at the wire boundary, and the overlay reads `.events` zero times. No two-shape client, no local→worker re-import path.
- **`Number(version)||1` vs `>=0` off-by-one; v0 resurrects deleted comments** (filed multiple times) — refuted: version is machine-generated starting at 1, upload rejects falsy version, the doc route 404s `v0` before the overlay loads, and the `999999` sentinel applies for all `at >= 999999` (intended "forever" semantics) — no resurrection.
- **Synthetic agent emoji corrupts reaction counts** — refuted: the injection is idempotent and re-derived every fold; the agent verdict is never persisted as a reaction record.
- **Local server lost-update race / non-atomic write** — refuted on the race (the only `await` is before `readJson`; the read-modify-write is fully synchronous on a single-threaded loop); the non-atomic-write durability point survives only as P3.
- **R2-then-meta-then-comments non-atomic → split-brain** (filed twice) — refuted: docs serve directly from R2 (not meta), the index is R2-`head`-guarded against advertising 404s, the meta write is *un*-try/caught so a real failure surfaces loudly, and full meta is re-sent every publish (self-healing). The proposed "meta first" fix would *reintroduce* the bug the verify guards against.
- **`reconcileAnchors` ignores text anchors → silent no-highlight** — refuted: orphaned text anchors get three visible signals (dashed border, "click to re-anchor" button, ghost marker) and text matching returns null rather than ever binding wrong.
- **`stampAids` cross-file cyrb53 "must match" contract** — refuted: the browser reads the stamped aid, never re-hashes; the two cyrb53 copies hash entirely different inputs and are compared only within their own tier.
- Plus refuted: `refreshComments` non-array parse (5xx returns HTML → existing catch handles it), `ensureMigrated` dirty-flag discard (lazy migration is idempotent + self-healing on read), reconcile redundant/oscillating events (version HTML is immutable → idempotent), `legacyToEvents` Object.keys ordering (insertion-order spec + set-union fold), `historyList` round-trip (pull is one-way by design; "lossless" = no comment dropped, correctly scoped), upload version string/number type mix (only caller sends a JSON number), and the doctor `set -uo` malformed-JSON path (only hex-constrained input is interpolated).

The common thread in the refutations: real code facts, false failure modes — claims that assumed a data flow that doesn't exist (local→worker comment push, browser-side aid hashing), misread a browser/runtime semantic (SameSite, CORS credentials, KV atomicity), or treated documented product behavior as a vulnerability.
